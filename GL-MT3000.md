# GL-MT3000 (Beryl AX) + Pi Zero 2 W — Synergy Guide

The Pi is the carrier access layer. The GL-MT3000 is the distribution and security layer.
Keep them independent — each works without the other.

## What the GL-MT3000 Does That the Pi Can't Easily Do

- **True dual-band WiFi 6** — dedicated 5 GHz radio keeps AP clients off the congested 2.4 GHz band entirely. In a hotel with 20 competing 2.4 GHz networks, this is a practical difference.
- **Serving many clients** — MT7981B with OFDMA + MU-MIMO handles concurrent clients cleanly. Pi Zero 2 W's shared Broadcom radio shows WiFi lag and drops even in light use.
- **2.5 Gbps WAN port** — future-proofed for gigabit hotel Ethernet that Pi's USB networking cannot saturate.
- **One-click VPN features** — kill switch, VPN cascading, Tor toggle, AdGuard Home — all in the web UI.
- **Native Multi-WAN failover** — switch between hotel WiFi, USB tether, and Ethernet automatically with zero scripting.
- **Captive portal handling** — dedicated workflow (MAC cloning, DNS rebind disable, browser redirect); Pi requires manually bridging the upstream network first.
- **USB 3.0 NAS** — plug an SSD, serve files at ~100 MB/s via SAMBA. Pi Zero 2 W has USB 2.0 OTG only.
- **WireGuard obfuscation (AmneziaWG)** — firmware v4.8.2+ hides WireGuard from DPI on restrictive networks.

## What the Pi Does That the GL-MT3000 Can't

- **Reliable iOS USB tethering** — ipheth/usbmuxd is the correct driver; GL-MT3000's tethering module has broken on every iOS major release through iOS 18.
- **Visible TTL bypass** — Pi has this dialed in and stable. GL-MT3000 can do TTL in theory but requires nftables syntax on newer firmware and has no test record against Visible 2025.
- **USB Ethernet gadget mode** — Pi exposes itself as a USB NIC to the MacBook (192.168.7.1). GL-MT3000 has no gadget mode.
- **Tailscale exit node** — Pi can route all Tailnet traffic out through Visible cellular. GL-MT3000 explicitly cannot be a Tailscale exit node.
- **Full Linux userland** — custom Python scripts, systemd services, kernel module loading, routing tables. GL-MT3000 is locked to OpenWrt ash + opkg.
- **Low-power operation** — Pi draws 0.5–1W; viable on USB battery banks for hours. GL-MT3000 needs 5–10W.

## Deployment Scenarios

### Scenario D — Pi Handles Carrier, GL-MT3000 Handles Encryption + WiFi 6 ⭐ Best Combined Setup

```
iPhone → [USB/ipheth] → Pi Zero 2 W
                           ↓ TTL=65 mangle, Visible bypass
                      [Pi AP: 10.3.141.0/24]
                           ↓ Wi-Fi 2.4 GHz or USB RNDIS cable
                      GL-MT3000 WAN
                           ↓ WireGuard tunnel → homelab/VPN
                      [GL-MT3000 AP: 192.168.8.0/24, WiFi 6, 5 GHz]
```

Clean separation: Pi = carrier access layer, GL-MT3000 = security + distribution layer. Clients behind GL-MT3000 get carrier bypass and VPN encryption without any per-device configuration. GL-MT3000 kill switch prevents unencrypted traffic if WireGuard drops.

**Gotcha:** Double NAT exists (Pi NATed, GL-MT3000 NATed). Fine for most use. For inbound connections use Tailscale or port-forward through both.

### Scenario G — USB RNDIS Backhaul (No WiFi-over-WiFi) ⭐ Pairs with Scenario D

```
Pi usb0 (g_ether gadget) → USB cable → GL-MT3000 USB port
                                            ↓ kmod-usb-net-rndis
                                       GL-MT3000 reads Pi as USB WAN
```

Avoids WiFi-over-WiFi backhaul. USB 2.0 RNDIS gives ~127 Mbps throughput — more than enough for Visible's ~30–50 Mbps. Pi side: `dtoverlay=dwc2` + `g_ether` (already configured). GL-MT3000 side: `opkg install kmod-usb-net-rndis`, add usb0 as WAN in LuCI.

**Note:** Pi's single OTG port is used for GL-MT3000 backhaul in this scenario — iPhone must use WiFi hotspot, not USB tethering.

### Scenario C — Parallel Tailnet (Independent but Both Visible on Tailnet)

```
Pi (<TAILSCALE_IP>)           GL-MT3000 (separate Tailnet node)
  - Visible cellular             - Hotel WiFi / any WAN
  - Subnet: 10.3.141.0/24       - Subnet: 192.168.8.0/24
  - Exit node                    - Uses Pi as exit node
```

Both devices on the same Tailnet. Pi advertises its cellular subnet and acts as exit node. GL-MT3000 routes its clients through Pi exit node for Visible-native routing. MacBook reaches both LANs without being physically connected to either.

**Gotcha:** GL-MT3000 cannot run Tailscale and WireGuard simultaneously. Pick one.

### Scenario A — Pi as Cellular WAN, GL-MT3000 Distributes to Many Clients

```
iPhone → Pi → [Pi AP 2.4 GHz] → GL-MT3000 Repeater (2.4 GHz STA)
                                       ↓
                                 GL-MT3000 AP (5 GHz, WiFi 6)
                                 [multiple clients]
```

Pi handles the carrier complexity, GL-MT3000 serves clients on clean 5 GHz. Pi's 2.4 GHz radio is consumed by the backhaul hop to GL-MT3000 — that's acceptable since GL-MT3000's 5 GHz is then fully dedicated to clients.

### Scenario B — GL-MT3000 Hotel WiFi Primary, Pi Cellular Failover

```
Hotel WiFi → [Repeater] → GL-MT3000 (primary WAN + Multi-WAN)
                                ↓ failover
                          Pi AP (Visible cellular)
                          [GL-MT3000 LAN: WiFi 6 to all clients]
```

GL-MT3000 handles hotel captive portals (its strongest use case). Pi stays on, idle, kicks in when hotel WiFi drops. Best for hotels where WiFi is usable but intermittent.

### Scenario F — GL-MT3000 WireGuard Obfuscation on Restrictive Networks

For corporate hotels or airports doing DPI to block WireGuard: GL-MT3000 firmware v4.8.2+ AmneziaWG hides the WireGuard fingerprint. Pi's standard Tailscale/WireGuard would be blocked in this environment — the GL-MT3000 adds an obfuscation layer the Pi cannot provide.

## What to Skip (Redundant or Overcomplicated)

| Combo | Why Skip |
|---|---|
| Both running independent VPN clients to the same exit | Double encryption, double overhead, zero gain |
| GL-MT3000 running Tailscale + WireGuard simultaneously | Explicitly unsupported; causes routing breaks |
| Pi AP → GL-MT3000 repeater → GL-MT3000 serving clients on same 2.4 GHz | Three hops on one radio; throughput will be terrible |
| Replacing Pi with GL-MT3000 entirely | Lose iOS USB tethering, USB gadget mode, exit node, Visible TTL bypass that works today |

---

## Technical Gotchas

1. **Tailscale + WireGuard conflict on GL-MT3000** — cannot run simultaneously. If GL-MT3000 runs Tailscale, its WireGuard client is disabled and vice versa. Pi has no such limitation.

2. **GL-MT3000 cannot be a Tailscale exit node** — confirmed in official docs. Only the Pi can serve as exit node.

3. **iOS tethering on GL-MT3000 is unreliable** — iOS 18 forum thread active as of 2026. Don't depend on it for WAN uplink redundancy.

4. **Hardware acceleration disables per-client rate limiting** — GL-MT3000's MT7981B NPU hardware NAT offload is mutually exclusive with tc/nftables per-client flow tracking. Choose one.

5. **Power when combined** — Pi (~1W) + GL-MT3000 (~7W) = ~8W total. Confirm your USB-C PD charger can supply both, especially if daisy-chained.

6. **USB RNDIS backhaul vs. USB tethering** — Pi has one OTG port. If using USB RNDIS to GL-MT3000 (Scenario G), iPhone must tether via WiFi hotspot, not USB cable.

## Quick Setup Reference (Scenario D + G)

**Pi side (already configured):**
- `dtoverlay=dwc2,dr_mode=peripheral` in `/boot/firmware/config.txt` ✅
- `g_ether` in `/etc/modules-load.d/g-ether.conf` ✅
- `usb0` static IP `192.168.7.1/24` in `/etc/dhcpcd.conf` ✅
- DHCP serving `192.168.7.2-100` via dnsmasq ✅
- TTL=65 mangle rules active ✅

**GL-MT3000 side:**
```bash
# SSH into GL-MT3000
ssh root@192.168.8.1

# Install RNDIS driver
opkg update && opkg install kmod-usb-net-rndis

# Plug Pi USB cable in, then in LuCI:
# Network → Interfaces → Add Interface → Name: PI_USB, Protocol: DHCP, Device: usb0
# Set as WAN in Multi-WAN with priority 2 (lower than hotel WiFi, higher than nothing)
```
